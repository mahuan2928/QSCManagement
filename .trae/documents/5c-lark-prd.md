## 1. プロダクト概要
本システムは、Lark Base `【最新】5C管理` を業務アプリ化する `5C巡検 Web システム` である。単に Base の表を Web に写すのではなく、`SV による評価 -> 問題指摘 -> 店舗の是正報告 -> SV の確認` を一気通貫で実行する。

- 対象 Base: `RADabvBDpavPzFsMmIHj7KOxpXc`
- Base URL: `https://lark-japan.jp.larksuite.com/base/RADabvBDpavPzFsMmIHj7KOxpXc?table=blk9wrEYf9B75Wcp`
- 対象ユーザー: `SV` / `店舗`
- 提供モード: `本番モード` / `Demo モード`

## 3. 絶対に守る業務ルール
### 3.1 最低遵守 Gate
このシステムの中心は点数計算ではなく、`最低遵守 22 / 22 OK` の Gate 制御である。

- `最低遵守点数 = 22 項目中 OK の件数`
- `店舗評価 = IF(最低遵守点数 = 22, "C評価", "D評価")`
- `最低遵守通過 = 最低遵守点数 === 22`

禁止事項:
- `85 点以上` を Gate 条件にしない
- `C評価以上` を Gate 条件にしない
- 空欄や未評価を `OK` とみなさない
- `22 / 22` 未満で `運営基準項目` / `価値創造項目` を解放しない

### 3.2 最低遵守未通過時の挙動
最低遵守に `NG` / `空欄` / `未評価` / `非 OK` が 1 件でもある場合:

- `運営基準項目` をロックする
- `価値創造項目` をロックする
- 未通過項目一覧を表示する
- `問題指摘` への起票を必須とする
- 店舗画面では是正タスクのみを見せ、後続評価の入口は見せない

### 3.3 最低遵守通過時の挙動
最低遵守が `22 / 22 OK` の場合のみ:

- `運営基準項目` を解放する
- `価値創造項目` を解放する
- SV は 5C 評価を継続できる

## 4. 評価体系
### 4.1 最低遵守項目
- 項目数: 22
- スコア: `OK = 1`、`NG / 空欄 / 未評価 = 0`
- 評価: `22 / 22 -> C評価`、`0 - 21 -> D評価`
- 役割: 通常の採点ではなく、後続工程を開放する前提 Gate

### 4.2 運営基準項目
- 項目数: 50
- 前提条件: 最低遵守 `22 / 22 OK`
- スコア: `50 項目中 OK 件数`
- 評価: `50 / 50 -> B評価`、`0 - 49 -> C評価`

### 4.3 価値創造項目
- 項目数: 20
- 前提条件: 最低遵守 `22 / 22 OK`
- スコア: `20 項目中 OK 件数`
- 評価: `0 - 9 -> B評価`、`10 - 17 -> A評価`、`18 - 20 -> S評価`

## 5. ユーザーと権限
### 5.1 SV
SV ができること:
- 店舗選択
- クール (`Q1` / `Q2` / `Q3` / `Q4`) 指定
- 評価日指定
- 最低遵守 22 項目の入力
- 最低遵守通過後の運営基準 / 価値創造入力
- NG 項目から問題指摘を作成
- 改善方法、改善期限の設定
- 店舗の是正報告確認

SV ができないこと:
- 最低遵守を飛ばして運営評価へ進む
- 最低遵守未通過のまま後続評価を強制解放する

### 5.2 店舗
店舗ができること:
- 自店舗に関する問題指摘の閲覧
- 改善期限、SV 指示の確認
- 是正コメント入力
- 改善後写真アップロード
- 是正完了報告

店舗ができないこと:
- SV の評価結果変更
- 最低遵守 / 運営基準 / 価値創造の採点変更
- 他店舗データ閲覧
- 是正を飛ばして問題を完了扱いにする

## 7. 認証・モード
### 7.1 本番モード
- Lark OAuth 2.0 ログインを利用する
- `user_access_token` で Base OpenAPI を呼ぶ
- フロントに `app secret` / `access token` を出さない
- 実際の Base 権限を尊重する
- API は認可後の範囲だけを返す

### 7.2 Demo モード
- ログイン不要
- 脱敏済み mock データのみ使用
- 実 Base は一切読まない
- 書き込みはローカル mock store のみ
- 画面と操作導線は本番相当

## 8. 是正クローズドループ
`問題指摘` が是正管理の主テーブルであり、最低遵守未通過時もここへ集約する。

主要フィールド:
- `アンケート集別`
- `店舗名`
- `第〇クール`
- `種別`
- `最低遵守指摘項目`
- `運営基準指摘項目`
- `価値創造指摘項目`
- `改善方法 ※SV設定`
- `改善期限`
- `写真`
- `コメント`

要求事項:
- 最低遵守の任意項目が `OK` でない場合、その項目を起点に `問題指摘` を作成する
- 店舗は `問題指摘` を確認し、コメントと改善後写真を返す
- SV は返却内容を確認し、是正完了を確定する

## 9. 受け入れ基準
- 最低遵守 `22 / 22 OK` のときのみ `運営基準` 入口が有効になる
- 最低遵守 `21 OK + 1 NG` のとき `運営基準` 入口は無効
- 最低遵守 `21 OK + 1 空欄` のとき `運営基準` 入口は無効
- 最低遵守未入力時 `運営基準` 入口は無効
- 最低遵守未通過時、未通過項目一覧を表示する
- 最低遵守未通過時、`問題指摘` を起票できる
- 店舗は自店舗の問題のみ閲覧できる
- 店舗は SV 採点を変更できない
- Demo モードは実 Base を参照しない
- 本番モードはフロントへ secret / token を露出しない

## 10. 一期の設計判断
- 是正フィードバックは `問題指摘` 拡張を優先する
- 理由: 一期は単一タスクに対する単一是正報告で十分であり、最短で運用投入できるため
- 二期で多段差し戻しや複数回報告が必要になれば、`是正フィードバック` 子テーブルを追加する
